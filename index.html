Background

Over the past few years, artificial intelligence has become increasingly present in our daily lives. Many have found new opportunities in this evolving technology, from getting general tasks done faster to supporting the creative process. Since AI became mainstream, image generators have flooded the internet. Nowadays, anyone is able to generate a picture from seemingly nothing.

However, most of these generators only let the user input a text description (a so-called "prompt") for the visual they want to create. While users might occasionally be able to change the model or aspect ratio, further control over the resulting image is often lacking. To counter that, the web app Latent Studio gives the user more possibilities to influence the final result. With easy-to-understand explanations accompanying the customizable inputs, the web app gives the user the opportunity to understand the process of image generation better, making the creative process more purposeful.

Abstract

Latent Studio is a web app created with the help of Google Colab, Gradio, Cursor, and Gemini. It is based on the architecture of Stable Diffusion 1.5 and focuses on text-to-image generation. To optimize the user experience, it exposes several easy-to-understand parameters within a clean interface, offering precise control over the resulting image.

Users can switch between the basic Stable Diffusion model, which sets a well-trained foundation, and the Dreamshaper model (created by Lykon) for more stylized creations.

On top of the model, the user can choose to add a smaller fine-tuned sub-model to get a specific look: the Low-Rank Adaptation, also known as a LoRA. How much the selected LoRA influences the result can be changed via the "LoRA Strength" slider, which dynamically appears after selection.

Following that, the user can write a prompt for what they want to see in the image and a negative prompt for what they want to avoid. If a LoRA is selected, the according trigger word will automatically be added to the prompt.

How many times the AI should improve the image can be changed with the "Steps" slider. Next to it, the user can define how strictly the AI should follow the prompt using the "CFG" (Classifier-Free Guidance) slider. The seed for each generation can be fixed for comparable results or randomized for more variety.

Lastly, if the user has a specific composition or object in mind they want to include as accurately as possible, they can enable ControlNet. Once enabled, a reference image can be uploaded. The amount of influence this image has on the result is controlled by the "Control Strength" Slider. For this specific ControlNet, the Canny Edge method is used, meaning that the edges of the reference image are extracted to guide the diffusion process. The resulting Canny Edge Map is output alongside the generated image and its metadata.

Approach

The general structure of the project was "vibe coded," mostly as a notebook within Google Colab with support from Google Gemini and Claude. Later in the process, I also used Cursor due to the limitations of Google Colab's free plan. The following sections explain the core code structure of the project:

  1. Environment Setup

    The purpose of this section is to prepare and install the external code libraries necessary to run machine learning models. Since image generation involves dense mathematical computations, the fastest available processor must be detected and configured. Because I temporarily transitioned to a local workspace using Cursor, the notebook is currently configured for macOS. It actively detects and activates Apple Silicon GPU (MPS) and falls back to NVIDIA GPU (CUDA) or CPU depending on the system hosting it.

  2. Model Initialization

    Here, the base model Stable Diffusion 1.5 and the Dreamshaper model are loaded directly from the AI community platform Hugging Face. Furthermore, the ControlNet is initialized: it loads a Canny Detector preprocessor along with the Canny ControlNet model. This allows the AI to capture outlines from a reference image and use them to guide the layout of the newly generated image.

  3. LoRA (Low-Rank Adaptation)

    Low-Rank Adaptation is a technique that lets you inject specific styles into a base model without having to retrain the entire network. They are highly effective for generating a specific object or aesthetic consistently. I previously trained two custom style LoRAs in a separate Google Colab notebook, resulting in .safetensors files that load in this section. In my dataset used for training, I paired every image with a descriptive text file. All of these files share one unique word in common; when this specific trigger word is present in the prompt, the LoRA activates.

    A LoRA is especially useful if the base model cannot generate an object or image style sufficiently on its own. During the conceptual phase, it is vital to compile a large, high-quality dataset. The images should have varying subjects but maintain a highly consistent overall style so the AI learns exactly what aesthetic rules to extract.

Low Poly LoRA

I wanted to add a style reminiscent of retro computer graphics with a charming, stylized touch. I was specifically drawn to characters and objects from early 2000s Nintendo games. I assembled a dataset of images fitting this exact aesthetic, featuring bright colors and hard edges resulting from low polygon limits. Every image in this dataset displays a single item or character on a clean white background. I used 48 images in total, pairing each with a simple descriptive .txt file to help the LoRA differentiate between the core style and the subject matter.

The trigger word defined for this LoRA is: lpyx7 style

Found Footage LoRA

For this LoRA, I wanted to see how accurately a high-contrast flashlight look could be transferred. Most of the training images featured a bright spot in one part of the image that faded into heavy darkness at the edges. I paid close attention to the color palette, as it was crucial for creating a spooky, atmospheric aesthetic. I also sourced different perspectives and subjects to ensure the style itself remained distinct from the objects depicted.

The trigger word defined for this LoRA is: FndF0ot4ge

Findings

Extra Descriptive Words in Prompts for Best Results

Low Poly LoRA

The Low Poly LoRA performed well with just the trigger word (lpyx7 style), but achieved optimal results when "low poly" was explicitly written in the prompt. Because the base model already understands the concept of "low poly," the text prompt reinforced the targeted style of the LoRA. Interestingly, generations with and without the LoRA using "low poly" looked quite similar, though the LoRA generation featured harsher, more defined shadows and a solid colored background reminiscent of the training data layout.

Visual Comparison: Low Poly LoRA Prompt Evaluations

Found Footage LoRA

Conversely, the Found Footage LoRA strictly requires the word "night" in the prompt for the desired effect to occur. Synonyms like "dark" or "darkness" only slightly desaturated the image without achieving the true flashlight aesthetic. Negative prompting for daylight also proved ineffective.

LoRA Strength Evaluation

Through testing, I determined that a strength of 1.3 for the Low Poly LoRA yields the cleanest results, making the style visible even without adding "low poly" to the prompt text. At a strength of 1.5, the structural integrity of the generation begins to degrade.

Difficulties & Challenges

1. Failed LoRA (The Bryce Render Concept)

In the beginning, I attempted to train a LoRA on the aesthetic of old CGI renderings—specifically from the legacy software Bryce 3D. While I loved the style, I realized that the appeal of these images relied heavily on complex, surreal compositions rather than simple shaders or lighting structures. The reference images were too disjointed for the model to effectively extract a cohesive style. This led me to pivot to the Low Poly LoRA, where I could build a cleaner, more uniform dataset.

Reference: Bryce Render Pack Examples & Generation Validations (Initial rifle generations)

2. Trigger Word Conflicts

Defining a unique trigger phrase presented unexpected hurdles. In my first attempt, I used the instance prompt low poly sks. Since "low poly" was part of the trigger, I couldn't test generations without the LoRA active while using those words, preventing an accurate baseline comparison. This prompted me to retrain the LoRA using the fully isolated trigger word lpyx7 style. Because the model has no pre-existing associations with "lpyx7," it does not muddy the generations with prior biases.

3. Cropping and Background Behavior

Since the Low Poly dataset consisted entirely of isolated assets centered on clean white backgrounds, the resulting generations often cropped the subject too tightly or scaled it too large. Utilizing the ControlNet Canny Edge function on a low setting helped anchor the composition nicely. Additionally, because the AI struggles with pure white, the generated backgrounds typically output light textures or subtle pastel tones instead of perfect white.
